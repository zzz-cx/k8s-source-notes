## 一、稳定性与 SRE 基础

### Q1. 什么是 SLA / SLO / SLI？区别是什么？

**一句话**：SLI 是「量什么」，SLO 是「目标值」，SLA 是「对外承诺（含违约后果）」。


| 概念      | 含义                             | 示例                       |
| ------- | ------------------------------ | ------------------------ |
| **SLI** | Service Level Indicator，可测量的指标 | 请求成功率、P99 延迟、消息送达率       |
| **SLO** | Service Level Objective，内部目标   | 成功率 ≥ 99.95%，P99 < 500ms |
| **SLA** | Service Level Agreement，对外合同   | 可用性 99.9%，未达标赔偿          |


**飞书举例**：IM 消息发送成功率、文档打开 P99 延迟、会议入会成功率都可作为 SLI。

---

### Q2. 99.9% 和 99.99% 可用性分别对应多少故障时间？


| 可用性          | 年故障预算     | 月故障预算     |
| ------------ | --------- | --------- |
| 99.9%（三个 9）  | ≈ 8.76 小时 | ≈ 43.8 分钟 |
| 99.99%（四个 9） | ≈ 52.6 分钟 | ≈ 4.38 分钟 |


**记忆**：每多一个 9，故障预算大约缩小 10 倍。

---

### Q3. MTTR、MTBF、MTTD 是什么？


| 指标       | 含义                                | 优化方向       |
| -------- | --------------------------------- | ---------- |
| **MTTD** | Mean Time To Detect，平均发现时间        | 监控、告警、巡检   |
| **MTTR** | Mean Time To Repair，平均恢复时间        | 预案、回滚、自动化  |
| **MTBF** | Mean Time Between Failures，平均故障间隔 | 架构改进、质量、容量 |


**面试答法**：降低事故影响，通常先优化 **MTTD**（早发现）和 **MTTR**（快恢复）；MTBF 更偏长期工程质量。

---

### Q4. 稳定性工作和普通后端开发的区别？

**答**：

- **后端开发**：关注功能实现、性能、业务逻辑正确性  
- **稳定性/SRE**：关注系统 **可靠运行、风险预防、故障快速恢复**，为 SLA 负责

稳定性同学的工作包括：监控告警、容量规划、发布保障、降级预案、应急止损、事故复盘、稳定性工具建设。本质是 **用工程手段让系统「抗得住、发现快、恢复快」**。

---

### Q5. 线上常见的稳定性风险有哪些？

1. **单点故障**：某组件挂了全链路不可用
2. **依赖超时/不可用**：下游慢或挂，拖垮上游
3. **容量不足**：CPU/内存/连接数/带宽打满
4. **发布缺陷**：新版本 bug、配置错误
5. **缓存/DB 异常**：慢查、主从延迟、缓存雪崩
6. **网络问题**：DNS、LB、跨机房链路
7. **配置/证书过期**：TLS 证书、密钥、限流阈值
8. **缺乏隔离**：无熔断限流，故障扩散（雪崩）

---

### Q6. 什么是单点故障（SPOF）？如何消除？

**SPOF**：系统中某一组件故障会导致整体不可用的设计。

**消除思路**：

- **冗余**：多副本、多 AZ/机房  
- **无单点架构**：主从切换、Leader 选举  
- **去中心化**：避免所有流量过单一网关/DB  
- **故障演练**：定期验证 failover 是否有效

---

### Q7. 故障域和爆炸半径是什么？

- **故障域（Fault Domain）**：故障影响范围的边界，如一台机器、一个机架、一个 AZ  
- **爆炸半径（Blast Radius）**：一次故障最多影响多大范围

**设计原则**：把组件分散在不同故障域；金丝雀发布、限流、熔断都是为了 **缩小爆炸半径**。

---

### Q8. 容量规划怎么做？

1. **基线**：日常 QPS、CPU、内存、带宽、连接数
2. **峰值**：大促、工作日早高峰、开学季等
3. **冗余**：通常留 30%～50% 余量
4. **压测**：验证在目标 QPS 下的延迟和资源
5. **弹性**：HPA、自动扩容、队列削峰

**预警信号**：CPU 持续 >70%、P99 延迟上升、队列堆积、连接池接近上限、错误率缓慢爬升。

---

### Q9. 混沌工程是什么？和压测的区别？

- **压测**：验证系统在 **高负载** 下能否撑住  
- **混沌工程**：在 **生产或类生产环境** 故意注入故障（杀 Pod、网络延迟、依赖不可用），验证系统的 **容错和自愈能力**

工具举例：Chaos Mesh、Litmus。前提是已有完善监控和快速回滚能力。

---

### Q10. 好的监控体系分哪几层？

```
业务层   → DAU、消息发送成功率、文档打开成功率
应用层   → QPS、延迟、错误率、线程池、GC
中间件层 → MySQL、Redis、MQ、ES 的连接数、慢查、堆积
基础设施 → CPU、内存、磁盘、网络、容器/Pod 状态
```

**原则**：能 **定位层级**、能 **关联告警**、避免只监控机器不监控业务。

---

### Q11. Metrics、Logs、Traces 分别解决什么问题？


| 类型          | 作用                | 典型工具               |
| ----------- | ----------------- | ------------------ |
| **Metrics** | 数值型时序指标，适合告警和趋势   | Prometheus、Grafana |
| **Logs**    | 离散事件记录，适合查具体错误    | ELK、Loki           |
| **Traces**  | 请求全链路耗时，适合定位慢在哪一跳 | Jaeger、SkyWalking  |


**组合使用**：告警用 Metrics → Trace 定位慢节点 → Logs 查具体异常栈。

---

### Q12. 告警设计原则？如何避免告警风暴？

**原则**：

- 告警必须 **可操作**（收到后知道干什么）  
- 分级：P0 立即响应 / P1 工作时间内 / P2 趋势关注  
- 基于 **用户影响** 而非单纯机器指标  
- 设置 **抑制和聚合**，避免同一根因触发上百条

**避免狼来了**：减少误报、合并相似告警、On-call 轮转、定期清理无效规则。

---

### Q13. 收到 P0 告警后，前 5 分钟做什么？

1. **确认影响面**：哪些服务/地域/功能？用户感知如何？
2. **止血优先**：回滚、降级、限流、切流量、扩容
3. **拉齐协作**：建事故群、指定 Incident Commander
4. **保留证据**：截图、监控链接、最近变更记录
5. **暂不深究根因**：先恢复服务，再排查

---

### Q14. 降级、熔断、限流的区别？


| 手段     | 含义             | 目的          |
| ------ | -------------- | ----------- |
| **限流** | 控制请求进入速率       | 保护自身不被打垮    |
| **熔断** | 依赖连续失败时暂时停止调用  | 防止拖垮上游、快速失败 |
| **降级** | 关闭非核心功能，返回兜底结果 | 保核心链路可用     |


**例子**：飞书文档编辑不可用，但只读/列表仍可用 → 降级；IM 发消息接口限流 → 限流；推荐服务连续超时则熔断 → 熔断。

---

### Q15. 预案应包含什么？

- **触发条件**：什么指标/场景启动  
- **执行步骤**：谁操作、操作什么（回滚版本、关功能开关、切 DB）  
- **影响说明**：降级后用户看到什么  
- **恢复条件**：何时回切  
- **联系人**：On-call、依赖方  
- **演练记录**：是否定期验证

---

### Q16. 事故复盘关注什么？为什么不能只追责？

**关注**：

- 时间线（何时发现、何时恢复）  
- 根因（技术 + 流程）  
- 影响范围（用户、数据、时长）  
- **改进项**（监控、预案、架构、测试）

**不只追责**：复盘目的是 **防止 recurrence**，Blameless Culture 鼓励如实暴露问题；追责会导致隐瞒信息。

---

### Q17. 飞书文档打开变慢，但 API 成功率仍 99.9%，怎么排查？

成功率不反映 **慢请求**。排查路径：

1. 看 **P95/P99 延迟** 是否上升
2. Trace 定位慢在：CDN / 网关 / 权限 / 存储 / 渲染
3. 查 DB 慢查、缓存命中率、对象存储延迟
4. 查最近发布、配置变更
5. 看是否特定地域/租户

---

### Q18. 依赖 Redis 的接口突然超时，先查应用还是 Redis？

**先快速分层**：

1. 同时看 Redis 监控（CPU、连接数、慢命令、内存、主从延迟）和应用监控（连接池、超时配置）
2. 若 Redis 指标异常 → 优先 Redis（大 key、热 key、持久化阻塞、网络）
3. 若 Redis 正常 → 查应用连接池是否打满、超时是否过短、是否有慢逻辑

**原则**：**先看监控仪表盘，再决定深入方向**，避免盲目改代码。

---

### Q19. 大促前稳定性要做什么？

- 容量评估与扩容  
- 全链路压测  
- 确认降级/限流/熔断开关可用  
- 检查监控告警、On-call 排班  
- 冻结非必要发布  
- 预案演练（DB 切换、机房故障）  
- 确认依赖方（短信、支付、CDN）容量

---

### Q20. 发布后 5% 错误率，回滚还是观察？


| 倾向回滚         | 可短暂观察                   |
| ------------ | ----------------------- |
| 核心链路、错误率持续上升 | 非核心功能、错误率稳定且低           |
| 影响大量用户       | 金丝雀小流量、有自动回滚            |
| 无明显缓解趋势      | 明确已知 benign 错误（如预期 404） |
| 无快速 fix 手段   | 有开关可一键降级                |


**SRE 原则**：**用户影响优先，快速止血**；有自动金丝雀+回滚机制时可观察分钟级。

## 二、网络与通信协议

### Q21. TCP 和 UDP 的区别？HTTP 基于哪个？


|     | TCP          | UDP       |
| --- | ------------ | --------- |
| 连接  | 面向连接         | 无连接       |
| 可靠性 | 可靠、有序、重传     | 不保证       |
| 开销  | 较大           | 较小        |
| 场景  | HTTP、RPC、数据库 | DNS、视频、游戏 |


**HTTP/1.1、HTTP/2 基于 TCP**；**HTTP/3 基于 QUIC（UDP）**。

---

### Q22. TCP 三次握手、四次挥手？

**三次握手（建立连接）**：

```
客户端 SYN → 服务端 SYN+ACK → 客户端 ACK
```

- 3 次：双方确认各自收发能力 + 同步初始序列号；2 次不够，无法确认客户端收到服务端应答

**四次挥手（关闭连接）**：

```
客户端 FIN → 服务端 ACK → 服务端 FIN → 客户端 ACK
```

- 4 次：TCP 全双工，每方向需单独关闭；服务端 ACK 和 FIN 常分开发送

---

### Q23. TIME_WAIT 过多有什么问题？

- 占用 **端口和连接表** 资源，高并发短连接场景可能 **无法建立新连接**  
- 处理：开启 `tcp_tw_reuse`（谨慎）、使用 **长连接**、连接池、调整内核参数、让客户端主动关闭时注意架构设计

---

### Q24. HTTP/1.1、HTTP/2、HTTP/3 主要差异？


| 版本       | 关键点                           |
| -------- | ----------------------------- |
| HTTP/1.1 | Keep-Alive、管线化有限；队头阻塞         |
| HTTP/2   | 二进制分帧、多路复用、头部压缩；仍受 TCP 队头阻塞影响 |
| HTTP/3   | 基于 QUIC(UDP)，解决传输层队头阻塞，连接迁移   |


---

### Q25. HTTPS 握手大致过程？证书作用？

1. 客户端 Hello（支持的加密套件）
2. 服务端返回 **证书**（含公钥）
3. 客户端验证证书（CA 链、域名、有效期）
4. 协商对称密钥，后续用对称加密通信

**证书作用**：证明服务端身份（防中间人）、传递公钥。

---

### Q26. 偶发连接失败，用什么命令排查？

```bash
ping / mtr          # 网络连通与丢包
nslookup / dig      # DNS 解析
curl -v             # HTTP 层状态码、TLS、耗时
telnet / nc         # 端口是否可达
ss -tunlp           # 本机连接状态
tcpdump             # 抓包
```

---

### Q27. DNS 解析慢会导致什么问题？

- 首次请求延迟高、超时  
- 依赖 DNS 的服务发现变慢  
- 排查：`dig +trace`、看 TTL、是否本地 DNS 缓存、是否 GeoDNS 解析到远节点

---

### Q28. 长连接和短连接的影响？

- **长连接**：减少握手开销，适合高频 RPC；但占用连接数，需心跳保活、超时管理  
- **短连接**：资源释放快，但高 QPS 下握手成本高

网关和后端连接池通常 **复用长连接** 提升性能。

---

### Q29. 502 / 504 / 499 通常表示什么？


| 状态码     | 含义                | 常见责任方      |
| ------- | ----------------- | ---------- |
| **502** | 网关收到无效上游响应        | 上游进程挂、协议错误 |
| **504** | 网关等待上游超时          | 上游慢、超时配置短  |
| **499** | 客户端主动断开（Nginx 扩展） | 客户端取消、前端超时 |


---

### Q30. 全链路延迟升高怎么分层定位？

按链路逐段对比 P99：**客户端 → CDN → LB → Gateway → 服务 A → 服务 B → DB/Cache**  

- 用 Trace 看每跳耗时  
- 若只有某一跳变慢，深入该层（GC、慢 SQL、锁竞争）  
- 若全链路均匀变慢，查网络、DNS、公共组件

## 三、负载均衡与应用服务器

### Q31. 负载均衡常见算法？


| 算法              | 说明                     |
| --------------- | ---------------------- |
| 轮询              | 依次分配                   |
| 加权轮询            | 按机器性能加权                |
| 最少连接            | 分给连接数最少的节点             |
| IP Hash / 一致性哈希 | 同一客户端固定到某节点，利于 Session |


---

### Q32. 四层 LB 和七层 LB 区别？


|     | 四层（L4）    | 七层（L7）              |
| --- | --------- | ------------------- |
| 依据  | IP + 端口   | HTTP 路径、Header、Host |
| 性能  | 更高        | 稍低，功能 richer        |
| 能力  | 转发        | 路由、重写、SSL 终结、A/B    |
| 例子  | LVS、F5 四层 | Nginx、Envoy、ALB     |


---

### Q33. Nginx / Envoy / HAProxy 了解吗？

- **Nginx**：高性能反向代理、静态资源、七层路由；配置简单  
- **Envoy**：云原生 L7 代理，动态配置、可观测性强，Service Mesh 数据面  
- **HAProxy**：专注负载均衡，L4/L7 性能优秀

---

### Q34. 健康检查失败阈值设太小/太大？

- **太小**：网络抖动导致频繁摘除，流量震荡  
- **太大**：坏节点长时间仍接流量，用户持续报错

**实践**：结合 **连续失败次数 + 间隔 + 恢复阈值（ hysteresis）** 配置。

---

### Q35. Session 粘滞的利弊？

**利**：有状态服务无需共享 Session 存储  
**弊**：节点不均衡、扩缩容丢 Session、故障时影响粘滞用户  

**更好方案**：Session 存 Redis，LB 无粘滞。

---

### Q36. 反向代理 vs 正向代理？

- **正向代理**：代客户端访问外网（翻墙、企业上网代理）  
- **反向代理**：代服务端接流量（Nginx 挡在前面），客户端不知道后端

---

### Q37. API Gateway 通常承担什么？

- 路由、鉴权、限流、熔断  
- SSL 终结、协议转换  
- 日志、监控、灰度发布  
- 聚合多个后端 API

---

### Q38. 一台后端变慢，LB 仍可能打流量过去？

**会**，默认轮询/最少连接不一定感知「慢」。  

**解决**：基于 **响应时间加权**、主动健康检查、熔断、自适应负载均衡；Service Mesh 可感知延迟。

---

### Q39. 蓝绿 / 金丝雀 / 滚动发布区别？


| 方式      | 特点         | 稳定性         |
| ------- | ---------- | ----------- |
| **蓝绿**  | 两套环境切换，回滚快 | 资源双倍，切换瞬间风险 |
| **金丝雀** | 小流量验证再全量   | 爆炸半径小，推荐    |
| **滚动**  | 逐步替换实例     | 资源省，中间版本混合  |

## 四、微服务架构

### Q40. 微服务的稳定性收益和风险？

**收益**：独立部署、故障隔离、团队自治、弹性扩缩  
**风险**：调用链变长、分布式复杂度、运维成本高、排查难、一致性难  

---

### Q41. 什么是服务发现？

服务实例动态变化时，客户端如何找到可用地址。  

**方式**：DNS、注册中心（Consul/Etcd/Nacos）、K8s Service + CoreDNS  

---

### Q42. 超时、重试、熔断、限流、隔离？


| 手段  | 作用                       |
| --- | ------------------------ |
| 超时  | 避免无限等待                   |
| 重试  | transient 失败可恢复；需幂等 + 退避 |
| 熔断  | 下游故障时快速失败                |
| 限流  | 保护自身                     |
| 隔离  | 线程池/舱壁，故障不扩散             |


**重试风险**：下游已超载时重试会 **放大流量**，需限重试次数 +  jitter。

---

### Q43. 雪崩效应？如何防止？

**雪崩**：局部故障 → 重试/阻塞 → 资源耗尽 → 全链路不可用  

**防护**：超时、熔断、限流、隔离、降级、避免同步长链、缓存、队列削峰。

---

### Q44. 什么是幂等？

同一操作执行多次，结果与执行一次相同。  

**重试必须幂等**：如用唯一 requestId、DB 唯一键、状态机校验。

---

### Q45. 分布式事务和最终一致性？

- **强一致分布式事务**：2PC/TCC，复杂、性能差，互联网少用  
- **最终一致性**：允许短暂不一致，通过消息、补偿、对账达到一致（常见）

---

### Q46. CAP 是什么？

分布式系统最多同时满足 **C（一致性）、A（可用性）、P（分区容错）** 中的两个；分区发生时常在 C 和 A 间权衡。  

- **注册中心**：通常 AP（可用优先，短暂不一致可接受）  
- **Redis 主从**：可能 AP；Cluster 分区时也有权衡  
- **强一致 DB**：CP 倾向

---

### Q47. A→B→C，C 挂了怎么避免拖垮 A？

- B 对 C：**超时 + 熔断 + 降级默认值**  
- A 对 B：同样保护  
- **异步化**、**缓存 C 的结果**  
- **舱壁隔离**：C 的调用独立线程池

---

### Q48. QPS 从 1k 到 10k，先动哪里？

1. 压测确认瓶颈（CPU/DB/Redis/网络）
2. **水平扩容** 无状态服务
3. **缓存** 热点读
4. **DB** 读写分离、索引、分库分表
5. **异步** 非核心逻辑走 MQ
6. **限流** 保护核心

---

## 五、存储与缓存

### Q49. MySQL 索引原理？何时失效？

**B+ 树**，加速查找，叶子节点链表支持范围查询。  

**可能失效**：对索引列函数/运算、隐式类型转换、前导模糊 `%abc`、不符合最左前缀、优化器判断全表扫描更优。

---

### Q50. 事务 ACID？隔离级别？

**ACID**：原子性、一致性、隔离性、持久性  

**隔离级别**：读未提交、读已提交、可重复读（MySQL 默认）、串行化  

---

### Q51. 主从复制和读写分离？

**主从**：Master 写 binlog，Slave 同步，**读扩展 + 高可用**  
**风险**：主从延迟导致 **读到旧数据**；需监控 `Seconds_Behind_Master`

---

### Q52. 慢 SQL 怎么排查？

```sql
EXPLAIN SELECT ...
SHOW PROCESSLIST;
-- 慢查询日志 slow_query_log
```

看：是否走索引、扫描行数、是否 filesort、临时表。

---

### Q53. 连接池为什么重要？

**复用连接**，避免频繁建连。打满现象：获取连接超时、接口大量超时、`Too many connections`。

---

### Q54. Redis 常见结构和场景？


| 结构     | 场景         |
| ------ | ---------- |
| String | 缓存、计数、分布式锁 |
| Hash   | 对象字段       |
| List   | 队列、时间线     |
| Set    | 去重、共同关注    |
| ZSet   | 排行榜、延迟队列   |


---

### Q55. 缓存穿透、击穿、雪崩？


| 问题     | 含义                    | 解决                 |
| ------ | --------------------- | ------------------ |
| **穿透** | 查不存在的数据，缓存和 DB 都扛     | 布隆过滤器、空值缓存         |
| **击穿** | 热 key 过期瞬间打穿 DB       | 互斥锁、逻辑过期           |
| **雪崩** | 大量 key 同时过期或 Redis 宕机 | 过期时间加随机、集群高可用、多级缓存 |


---

### Q56. 缓存与 DB 一致性？

常用 **Cache Aside**：

- 读：先缓存，miss 读 DB 再写缓存  
- 写：先更新 DB，再 **删缓存**（而非更新缓存，避免并发乱序）

高一致场景：延迟双删、订阅 binlog、分布式锁。

---

### Q57. Redis 单线程为什么快？

- 纯内存  
- IO 多路复用  
- 高效数据结构

**阻塞操作**：大 key 删除、`KEYS` *、持久化 fork、复杂 Lua 长时间执行。

---

### Q58. 热点 Key 怎么处理？

- 本地缓存 + Redis  
- 热 key **拆分**（加随机后缀分散到多个 key）  
- 读写分离、Proxy 层聚合  
- 限流

---

### Q59. 缓存命中率从 95% 掉到 60%？

查：是否 **大量新 key**、缓存 **提前过期**、Redis **内存淘汰**、**热 key 失效**、应用 **缓存逻辑 bug**（key 变了）、DB 流量是否对应上升。

---

### Q60. DB CPU 高但 QPS 没涨？

可能：**慢 SQL**、锁等待、**大事务**、索引失效导致全表扫描、后台任务（统计、备份）、主从延迟堆积、病毒式定时任务。

---

## 六、安全

### Q61. 认证和授权的区别？

- **认证（Authentication）**：你是谁 — 登录、Token、证书  
- **授权（Authorization）**：你能做什么 — RBAC、权限列表

---

### Q62. JWT 原理和风险？

**结构**：Header.Payload.Signature，服务端用密钥验签，无状态。  

**风险**：泄露即被盗用、难主动失效（需黑名单/短过期）、Payload 不加密仅 Base64（勿存敏感信息）。

---

### Q63. SQL 注入、XSS、CSRF？


| 攻击     | 含义          | 防护                               |
| ------ | ----------- | -------------------------------- |
| SQL 注入 | 恶意 SQL 拼接入参 | 预编译、参数化                          |
| XSS    | 注入恶意脚本      | 转义、CSP、HttpOnly Cookie           |
| CSRF   | 跨站伪造请求      | Token、SameSite Cookie、Referer 校验 |


---

### Q64. HTTPS 中间人攻击？证书如何防？

攻击者截获通信冒充服务端。**CA 签发的证书** 让客户端验证服务端公钥真实性；证书链校验失败则浏览器警告。

---

### Q65. 微服务间如何做身份校验？

- **mTLS**（双向 TLS，Service Mesh 常见）  
- **Service Token / JWT** 服务间传递  
- K8s **ServiceAccount** + RBAC

---

### Q66. 密钥为什么不能进代码仓库？

泄露不可撤销、扫描机器人专门搜 GitHub、合规要求。**应使用** 密钥管理系统、环境变量、Vault、K8s Secret（并启用加密 at rest）。

---

## 七、K8s / Linux / 基础设施

### Q67. Pod、Deployment、Service 是什么？


| 资源             | 作用                           |
| -------------- | ---------------------------- |
| **Pod**        | 最小调度单元，一个或多个共享网络的容器          |
| **Deployment** | 管理无状态 Pod 副本、滚动更新、自愈         |
| **Service**    | 稳定访问入口，ClusterIP + 负载均衡到 Pod |


---

### Q68. Pod 一直 Pending？

- **调度**：资源不足、nodeSelector/亲和性、**污点无 toleration**、PVC 未绑定  
- **未调度但已绑定**：镜像拉取失败、准入拒绝  
- 排查：`kubectl describe pod` 看 Events

---

### Q69. Liveness 和 Readiness 探针区别？


| 探针            | 失败后果                  |
| ------------- | --------------------- |
| **Liveness**  | 认为容器死，**重启**          |
| **Readiness** | 从 Service **摘流量**，不重启 |


**配错**：Liveness 过敏感 → 反复重启；Readiness 不配 → 未就绪仍接流量。

---

### Q70. HPA 是什么？

**Horizontal Pod Autoscaler**，根据 CPU/内存/自定义指标自动增减 Pod 副本，应对流量波动，是稳定性 **容量弹性** 手段。

---

### Q71. ClusterIP / NodePort / LoadBalancer？


| 类型           | 访问方式               |
| ------------ | ------------------ |
| ClusterIP    | 集群内虚拟 IP           |
| NodePort     | NodeIP:30000-32767 |
| LoadBalancer | 云厂商 LB → NodePort  |


---

### Q72. cgroup 是什么？为什么 driver 要一致？

**cgroup**：Linux 资源隔离（CPU/内存）。kubelet 和容器运行时各自管理 cgroup 路径；**driver 不一致**（systemd vs cgroupfs）会导致路径冲突、kubelet 启动失败或资源限制失效。

> 详见笔记：[04-node/kubelet/cgroup.md](../04-node/kubelet/cgroup.md)

---

### Q73. Linux 如何看 CPU、内存、磁盘、网络？

```bash
top / htop          # CPU、内存进程
free -h             # 内存
df -h               # 磁盘
iostat              # 磁盘 IO
ss -s / ss -tunlp   # 网络连接
vmstat 1            # 整体资源
```

---

### Q74. 进程 CPU 高、内存泄漏怎么定位？

**CPU**：`top` 找进程 → `perf`/`pprof`/火焰图 → 是否 GC 频繁、死循环、正则灾难  
**内存**：`pmap`/heap dump → 是否缓存未释放、连接泄漏、大对象  

---

### Q75. 容器和虚拟机的区别？


|     | 虚拟机            | 容器                     |
| --- | -------------- | ---------------------- |
| 隔离  | 硬件级 Hypervisor | 进程级 namespace + cgroup |
| 启动  | 分钟级            | 秒级                     |
| 镜像  | 完整 OS          | 应用 + 依赖                |
| 密度  | 较低             | 较高                     |


---

## 八、工具与自动化

### Q76. 监控/日志工具？

- **Prometheus + Grafana**：指标与大盘  
- **ELK / Loki**：日志  
- **Jaeger / SkyWalking**：链路追踪  
- **Alertmanager**：告警路由

---

### Q77. CI/CD 和稳定性的关系？

CI：自动化测试，减少缺陷上线  
CD：自动化发布，配合金丝雀、回滚、变更窗口，降低人为失误。

---

### Q78. 如何用工具提升排障效率？

举例：一键拉取 Pod 日志+Events+最近变更；发布前自动检查副本数/探针/资源 limit；事故时间线自动从告警系统生成。

---

### Q79. 设计「发布检查工具」检查什么？

- 镜像 tag 是否固定（非 latest）  
- 是否配置 readiness/liveness  
- 资源 requests/limits  
- 副本数 ≥ 2（非单点）  
- 是否有对应监控告警规则  
- 是否有关联降级开关

---

### Q80. Infrastructure as Code？

用代码管理基础设施（Terraform、Pulumi、Helm），**可版本化、可 Review、可重复**，减少配置漂移。

## 九、综合设计题（答题框架）

### Q81. 设计飞书消息发送链路的监控告警

**SLI**：发送成功率、P99 延迟、端到端送达率  

**Metrics**：网关 QPS/错误率、MQ 堆积、DB 写入延迟、推送第三方成功率  

**Logs**：失败 messageId、错误码  

**Trace**：客户端 → API → 消息服务 → 存储 → 推送  

**告警**：成功率 < SLO、P99 超阈值、队列堆积 > N、错误码突增  

---

### Q82. 核心服务依赖 3 个下游，如何设计降级？

- 区分 **强依赖 / 弱依赖**  
- 弱依赖：超时短 + 熔断 + 返回默认值  
- 强依赖：重试有限 + 排队 + 清晰错误提示  
- 全局 **降级开关**：关闭推荐、缩略图等非核心  
- 监控每个依赖的 **错误预算**

---

### Q83. 为新微服务制定 SLI/SLO

1. 明确 **用户旅程**（如「创建文档」）
2. 选 SLI：成功率、延迟、吞吐
3. 定 SLO：参考依赖方要求和历史数据（如 99.95% + P99<200ms）
4. 配 **错误预算** 驱动发布节奏
5. 上线后 **迭代** SLO

---

### Q84. 设计一次故障演练（GameDay）

1. 定场景：单 AZ 故障、DB 主库挂、依赖超时
2. 非生产或可控生产窗口
3. 注入故障（Chaos Mesh / 手动断网）
4. 观察：告警是否触发、On-call 响应、预案是否有效
5. 复盘改进项

---

### Q85. 大版本发布前稳定性 Checklist

- 压测通过  
- 金丝雀/灰度方案  
- 回滚脚本验证  
- 监控告警就绪  
- 降级开关测试  
- On-call 排班  
- 依赖方对齐  
- 变更公告与 freeze 窗口

---

## 十、行为面

### Q86. 说一个排查过最复杂的问题（STAR 模板）

- **S** 场景：现象、影响  
- **T** 任务：你的角色  
- **A** 行动：分层排查、协作、止血  
- **R** 结果：恢复时间、后续改进（监控/预案/代码）

---

### Q87. 开发赶上线 vs 稳定性冲突？

- 用 **数据** 说明风险（无监控、无回滚的影响）  
- 推动 **最小稳定性底线**（探针、回滚、核心监控）  
- 支持 **分阶段上线**（金丝雀）平衡速度与风险  
- 事后补债务，不永久妥协

---

### Q88. 如何学习新技术？最近在学什么？

**示例答法**：通过官方文档 + 动手实验 + 源码阅读结合。最近在系统学习 **Kubernetes 源码**（kubelet、cgroup、调度、Service 网络），并整理学习笔记和架构图，理解容器编排背后的控制循环和资源隔离机制，这和稳定性工作中的故障排查、K8s 运维直接相关。

---

### Q89. 为什么想做稳定性而不是纯业务开发？

对 **系统可靠运行** 更感兴趣；稳定性工作覆盖架构、网络、存储、发布、应急， **视野更广**；解决的是「千万用户能否正常使用」的问题，成就感强；和开发互补，能推动工程文化改进。

---

### Q90. 飞书稳定性挑战可能在哪？

- **实时协作**：IM、文档协同，对 **延迟和一致性** 要求高  
- **峰值明显**：工作日早晨、开学季、大活动  
- **链路长**：客户端 → 网关 → 多微服务 → 存储 / 推送  
- **多租户**：隔离与 noisy neighbor  
- **全球化**：跨地域网络、合规

